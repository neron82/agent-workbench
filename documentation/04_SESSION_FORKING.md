# Consolidated Session Forking Specification

Status: Finalized product-layer fork contract with user decisions applied.

## 1. Core rule
Chat, Research, and Work are distinct session types.
A session may not mutate from one type to another in place.
A type change requires a structured fork.

## 2. Fork triggers
Allowed triggers:
- user explicit fork request
- orchestrator-initiated fork after explicit/strong signal
- replay or retry flow

Automatic fork suggestions:
- conservative only
- triggered by strong project/research/work signals
- never silent

## 3. Required fork payload
Every fork must capture:
- parent_session_id
- child_session_id
- new_session_type
- fork_reason
- initial_summary
- decisions_so_far
- assumptions
- open_questions
- relevant_artifacts
- proposed_agent_assignments
- checkpoint_json
- initiated_by
- created_at

## 4. Storage decision
Fork metadata lives in product-layer fork tables inside `workbench.db`.
It does not live only in Hermes session tables.

## 5. Summary generation decision
Fork summaries are generated by a dedicated Consolidator/Summarizer function.
For important forks, use orchestrator / stronger model selection.

## 6. Bootstrap context decision
Internal product representation:
- `fork_context`

Model injection behavior:
- inject fork context as system-level context when calling models

This preserves product semantics without requiring providers to support a custom role directly.

## 7. Inheritance rules
Default inheritance:
- summary-based only
- carry forward structured decisions, assumptions, open questions, artifacts
- do not carry raw transcript unless explicitly requested

Raw transcript inheritance:
- explicit opt-in only
- must be visible to reviewer as a heavier context path

## 8. Gateway support scope
Messaging gateways support:
- limited fork creation
- auto-generated summary only

Full editable fork forms belong to:
- Web
- CLI
- ACP

## 9. Checkpoint decision
Fork checkpoints use standardized versioned JSON.
Minimum shape:
- version
- source_session_id
- source_message_offset or source_turn_id
- token_position optional
- prompt_hash optional
- artifact_refs optional

## 10. Validation rules
Reject fork if:
- parent session does not exist
- target type is invalid
- summary is empty
- structured fields exceed configured inheritance bounds
- child session cannot be created atomically with fork metadata

Warn if:
- artifact references are unresolved but expected
- parent is stale relative to fork request
- summary confidence is low

## 11. Reviewability rules
- child session must clearly display parent linkage
- fork summary must be inspectable separately from child-generated work
- reviewer must be able to see which context was inherited and which work was produced after the fork

## 12. Relationship to Hermes
Reused:
- `parent_session_id`
- existing session persistence
- ACP/session lineage surfaces where helpful

Added by product layer:
- typed session semantics
- structured fork payload
- fork tables in `workbench.db`
- standardized checkpoint schema
- summary-generation contract
- conservative suggestion policy
