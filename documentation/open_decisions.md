1. Use a separate product-layer SQLite database, workbench.db, for channels, sessions, messages, events, forks, runs, artifacts, and routing metadata.

2. Treat Hermes and Opencode as harnesses/backends, not as the product truth layer.

3. Manage Opencode as one harness instance per HarnessRun / worker task. Do not use one global shared Opencode server as the default.

4. Use Hermes subagent delegation as the default Research orchestration pattern. Kanban is a status/visualization layer, not the primary autonomous orchestration mechanism.

5. Add a product-layer permission model for destructive/sensitive actions. It may be minimal in Phase 3 but must exist in the architecture.

6. Default routing is user → orchestrator → worker. Direct worker dispatch is allowed only through explicit @agent addressing or UI target selection.

7. Define @all as broadcast to active non-execution discussion participants. It must not trigger execution workers by default.

8. SSH-backed runs must track remote process identity and provide best-effort cancel/reap. Long-term contract requires reconnect/status/reap support.

9. AgentProfiles are global by default, with workspace/profile-level overrides later.

10. Effective toolset is negotiated from AgentProfile permissions, Harness capabilities, and Session policy.

11. Kanban-backed workers use block-and-re-spec as the default steering model. Mid-run steering is harness-specific.

12. Agent profile changes during a session create a new AgentInstance or versioned binding.

13. Function vocabulary is closed for MVP but should be designed for later registry extension.

14. Capabilities stay mostly toolset-driven for MVP, with minimal first-class model capability metadata.

15. Fork metadata lives in product-layer fork tables.

16. Fork summaries are generated by a dedicated Consolidator/Summarizer function. Use orchestrator/strong model for important forks.

17. Store fork bootstrap context as custom fork_context internally; inject it as system-level context when calling models.

18. Automatic fork suggestions should be conservative and triggered only by explicit or strong project/research/work signals.

19. Messaging gateways support limited fork creation with auto-summary only. Full editable fork form belongs in Web/CLI/ACP.

20. Fork checkpoints use standardized versioned JSON.

21. Persist routing metadata for every message/event.

22. Every session/run/board should have workspace_id/tenant_id, with a default single-user workspace.

23. Workbench maintains its own HarnessRun metadata layer. Opencode run history is not sufficient as the product history.

24. Replay equivalence means equivalent final state and reviewer-judged outcome, not identical tool-call sequence.

25. Pause remains a future capability flag. MVP uses steer, stop, and cancel.

26. Sensitive Work sessions may escalate to per-tool confirmation even if auto-approve is enabled.

27. Decomposition approval defaults to dependency-order approval, with batch approval for low-risk batches and explicit tree UI later.
